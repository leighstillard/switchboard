# Switchboard v1.1 — Feature Brief

**Status:** Brief, rev 2 (post template-inventory)
**Owner:** Leigh Stillard / Format5
**Last updated:** 2026-05-03
**Builds on:** `design.md` rev 2 (v1 design), `test-plan.md`

> **Rev 2 changelog (2026-05-03):** Reworked Feature 1a around dedicated Slack block types for agentic rendering — `task_card`, `plan`, `markdown`, `card`, `alert`, `carousel`, `feedback_buttons`. These obsolete the custom Block Kit composition sketched in rev 1, simplify the rendering layer significantly, and remove the need for a custom mrkdwn converter. Feature 1b (tool approval) now anchored on the dedicated `approval.json` and `approval_with_carousel.json` templates. Added Feature 1c (terse tool descriptions) — agent-supplied or heuristic, ≤8 words target, drift-monitored. New subsection on render-directives — the agent-cooperation protocol for triggering specific templates from agent output. Slack-go library coverage confirmed for all block types used; rendering verification deferred to live deployment with text fallbacks as the safety net.

---

## Scope

Two features, both addressing concrete v1 gaps surfaced in design review:

1. **Block Kit upgrade** — three pieces: (1a) richer rendering for agent progress and notifications, (1b) interactive components for tool approval (gated on jcode protocol work), and (1c) terse tool descriptions for at-a-glance scanning.
2. **LLM-based notification router** — fallback for ambient events that don't match a deterministic rule, with a confidence threshold and audit trail.

Everything else in the v1.1 candidates list (`session_state` persistence, `file_shared`, slash commands, etc.) is explicitly *not* in v1.1 unless it falls out for free during this work.

## Why these two now

Block Kit and LLM routing were the two gaps Leigh identified as actively painful, not the longest deferral list. They're also independent of each other (different code paths, different test surfaces), so they can ship together or sequentially without coupling.

Block Kit interactive buttons are blocked on a jcode protocol PR — see "Dependencies" below. The rendering portion of Block Kit is not blocked and can ship independently.

## What v1.1 is *not*

- A redesign. The v1 architecture (jcode socket → coalescer → per-channel queue → Slack) is unchanged.
- Schema breaking. v1.1 adds tables and columns; no migrations destroy data.
- A behaviour change for existing v1 features. Routing, sessions, restart guarantees, audit log, file handling — all unchanged.

If a v1.1 change requires altering established v1 contracts, it gets bumped to v2 and discussed there.

## Feature 1: Block Kit upgrade

### Background — what changed since rev 1

Slack has shipped dedicated block types for agent and assistant rendering since the v1 design was written. Rather than approximating progress messages and approval prompts using compositions of `header` + `section` + `actions` + `context` blocks (and a custom mrkdwn converter), v1.1 uses purpose-built blocks where they exist. This is a significant simplification — what was sketched as bespoke layout work in rev 1 becomes "pick the right template per intent and map fields into it."

The block types we lean on:

- **`task_card`** — single streaming task with interleaved text + tool calls. Slack's "timeline mode" for thinking-steps. Has native `task_id`, `status: in_progress|complete`, `details`, `output`, `sources` fields.
- **`plan`** — multi-step plan with grouped tasks, each with its own status, details, output, and sources. "Plan mode" for thinking-steps. Use when the agent has a sequence of distinct steps.
- **`markdown`** — native GitHub-flavored markdown rendering. Tables, fenced code with syntax highlighting, task lists (`- [x]`/`- [ ]`), blockquotes. **Replaces the custom mrkdwn converter from v1.**
- **`card`** — title + subtitle + body + actions, all in one block. Foundation of `notification_with_actions`, `notification_with_deploy_preview`, `approval`.
- **`alert`** — first-class `level: success|error|warning` with appropriate visual treatment. Replaces the custom warning variant of notification-with-actions.
- **`carousel`** — swipeable card sequence. Used in `approval_with_carousel` (batch tool approvals) and `deep_research_result` (citations).
- **`feedback_buttons`** — native thumbs-up/down with structured `action_id` and value. Used for the LLM-routing feedback loop.
- **`context_actions`** — container for inline action elements like feedback_buttons.

All of these are present in `slack-go/slack` with constructors and tests; no upstream contribution required. Reference templates are committed at `github.com/leighstillard/slack-blockkit-templates`.

### Three distinct pieces

**1a. Rendering upgrade (no protocol dependency).**

A render-directive system in the bridge picks the right template per output type and populates fields from the underlying event stream. The mapping:

| Communication intent | Template / block | Source event(s) |
|---|---|---|
| Live agent progress (in-thread, streaming) | `task_card` block | jcode `TextDelta`, `ToolStart`, `ToolDone`, `MessageEnd` events from the coalescer |
| Multi-step agent plan / dispatch | `plan` block | Agent emits a `render: plan` directive (see §1a Render directives) |
| Completion report at end of turn | `markdown` block with `- [x]`/`- [ ]` task lists | jcode `Done` event with the coalesced tool summary |
| Detailed brief with citations | `deep_research_result` shape (section + markdown + carousel of source cards) | Agent emits a `render: brief` directive |
| Webhook notification (no image) | `card` block + `context` footer | webhook ingest, no `image_url` field |
| Webhook notification with actions | `card` block with `actions` array | webhook ingest where the rule template includes action buttons |
| Webhook notification with image | `card` block with `hero_image` (deploy-preview shape) | webhook ingest with `image_url`, or jcode `GeneratedImage` |
| Q&A poll | `poll` shape (section + per-option vote cards) | Agent emits a `render: poll` directive |
| Onboarding | `simple_onboarding` shape | First mention in unconfigured channel |
| Backlog / tickets | Multiple `task_card` blocks, or section-rows lifted from `ticket_list` | Agent emits a `render: tickets` directive |
| Tasks / to-dos | Section-rows lifted from `to_do_app` | Agent emits a `render: todos` directive |
| LLM routing feedback | `feedback_buttons` inside `context_actions` block | Appended automatically to LLM-routed messages (see Feature 2) |
| Errors / warnings | `alert` block with `level: error/warning/success` | Bridge errors, jcode `Error` events, retry-exhausted webhooks |
| Multiprovider fallback footer | `context` block (italic mrkdwn line) | Coalescer's captured `UpstreamProvider` value |

The internal architecture is straightforward: the coalescer and webhook router each emit a render-directive struct; the rendering layer is a single function per directive type that produces the JSON for the appropriate block. No mrkdwn translation layer (the `markdown` block renders our markdown directly), no custom progress-message composition. Most rendering functions are 20-50 lines.

**Render directives (agent cooperation protocol).** For templates that need richer-than-text input from the agent (plans, briefs, polls, tickets, todos), the bridge teaches the agent — via system-prompt addendum — to emit fenced JSON blocks with a recognised `render` field:

```
Agent emits in its TextDelta stream:

```switchboard
{
  "render": "plan",
  "title": "Migrate Postgres to Aurora",
  "tasks": [
    {"id": "1", "title": "Schema dump", "status": "complete"},
    {"id": "2", "title": "Apply DBAssure POC", "status": "in_progress"},
    {"id": "3", "title": "Verify row counts", "status": "pending"}
  ]
}
```
```

The bridge intercepts these fenced blocks in `TextDelta` events, validates against the directive schema, removes them from the visible text stream, and replaces with the rendered Block Kit equivalent. Unrecognised or invalid directives fall through as visible code blocks (so debugging is easy and bad output is loud).

The directive vocabulary for v1.1: `plan`, `brief`, `poll`, `tickets`, `todos`. Each has a JSON schema in `internal/render/schemas/`. The vocabulary is documented in the system-prompt addendum that's prepended to every jcode session, and in `docs/render-directives.md` for human reference.

**Block-type fallback strategy.** Every `chat.postMessage` and `chat.update` call sets a meaningful `text` parameter — Slack's official fallback mechanism for clients that can't render a particular block type. The fallback is a one-line plain-text summary of the message: for a `task_card`, the summary is the task title plus current status; for an `alert`, the alert level plus first sentence; for a `plan`, the plan title plus task counts. If a client (mobile, older Slack desktop, screen reader) can't render the block, it shows the fallback. We don't need to test rendering across clients before shipping; we just need the fallback to be useful when rendering fails.

This is a redo of M3's rendering layer plus the new directive interceptor. ~3-4 days of focused work given the dedicated block types do the heavy lifting.

**1b. Interactive components for tool approval.**

When jcode emits a `ToolApprovalRequest` (see Dependencies), the bridge posts an approval message anchored on Slack's dedicated approval templates:

- **Single tool call requiring approval:** post the `approval.json` shape — a `card` block with title, subtitle, body, primary `Approve` and danger `Deny` buttons. Optional `Allow all <tool_name>` button as a non-styled secondary action.
- **Multiple tool calls in flight (jcode emits batch approvals):** post the `approval_with_carousel.json` shape — header + section explaining the batch + `carousel` of approval cards, one per tool call. User swipes through and decides each.

User clicks a button → Slack posts an interactivity payload to switchboard's interactivity URL → switchboard validates the signing secret, looks up the pending approval(s) by `tool_call_id`, sends `ToolApprovalResponse` back over the jcode socket, edits the message via `chat.update` to show resolution status (the card's `actions` array is replaced with a `context` block: "✓ Allowed by @leigh at 14:32"), the buttons disappear.

Pending approvals are persisted in SQLite — if the bridge restarts mid-approval, the buttons remain functional after reconnect, because Slack's interactivity payload includes the original `action_id` and we look up state by that. Approval requests time out after a configurable interval (default 10 minutes) → automatic deny, posted as an updated message.

### State additions

```sql
-- Pending tool-approval requests. Survives bridge restart.
CREATE TABLE pending_approvals (
  tool_call_id    TEXT PRIMARY KEY,
  channel_id      TEXT NOT NULL,
  thread_ts       TEXT NOT NULL,
  message_ts      TEXT NOT NULL,        -- the approval message with the buttons
  jcode_session   TEXT NOT NULL,
  tool_name       TEXT NOT NULL,
  tool_input_json TEXT NOT NULL,
  batch_id        TEXT,                  -- nullable; set when carousel-grouped
  requested_at    INTEGER NOT NULL,
  expires_at      INTEGER NOT NULL,
  status          TEXT NOT NULL,         -- 'pending', 'allowed', 'denied', 'allow_all', 'expired'
  resolved_by     TEXT,                  -- Slack user_id of who clicked
  resolved_at     INTEGER
);

CREATE INDEX idx_approvals_batch ON pending_approvals(batch_id) WHERE batch_id IS NOT NULL;
CREATE INDEX idx_approvals_expires ON pending_approvals(expires_at) WHERE status = 'pending';

-- Per-session "allow all" persistence so subsequent calls
-- to the same tool in the same session skip the prompt.
CREATE TABLE session_tool_grants (
  jcode_session   TEXT NOT NULL,
  tool_name       TEXT NOT NULL,
  granted_at      INTEGER NOT NULL,
  granted_by      TEXT NOT NULL,         -- Slack user_id
  PRIMARY KEY (jcode_session, tool_name)
);
```

### New Slack scopes

- The interactivity URL endpoint must be configured in the Slack app manifest. This is a one-time manifest update, not a new scope.
- No additional bot scopes required — `chat:write`, `chat:write.customize`, and `chat:write.public` (already in v1) cover all rendering and editing needs.

### Configuration

```toml
[approvals]
enabled = true                          # off if jcode protocol PR not merged
default_timeout_seconds = 600           # 10 min
require_for_tools = ["Bash", "Edit", "Write", "MultiEdit"]
auto_allow_for_tools = ["Read", "Glob", "Grep"]
allow_carousel_threshold = 2            # post as carousel when N pending; single card below that

[render]
# Maximum size of a markdown block before splitting into multiple messages.
# Slack's hard limit is around 12K chars, but render quality degrades earlier.
markdown_split_chars = 8000

# When agent emits a render directive that fails schema validation,
# we leave it as a visible code block so the failure is loud.
strict_directive_validation = false  # if true, invalid directives are dropped silently
```

The `require_for_tools` / `auto_allow_for_tools` lists let operators tune which tools prompt — the v1 model of "agent runs everything autonomously" survives by default, with approval gating opt-in per tool.

### Open questions for the design phase

- **Does the Allow button add an entry to a session-level allowlist, or is it per-call only?** Spec sketches both via `session_tool_grants`. Probably per-call by default, with `Allow all <tool>` writing the grant. Confirm in design.
- **What happens if multiple users click different buttons?** Probably first-write-wins; the table's status field transitions exactly once. UX choice: edit message to show "✓ Allowed by @leigh (you), Denied attempt by @aaron 0.3s later".
- **Display of large tool inputs in approval cards.** A Bash command of 200 chars is fine inline; a 10KB diff is not. The `card.body` field has a 3000 char soft limit. Truncate with "show more" link to a Slack snippet, or use a `markdown` block as a sibling to the card? Defer to design.
- **Approval scope across threads.** Probably scoped to a single session/thread. A grant in #impex-scraper does not extend to #slopwatch.
- **Carousel vs separate cards for batch approvals.** Carousel is swipeable on desktop, vertical-stack on mobile. Worth checking whether mobile users find vertical-stack of 5 approval cards overwhelming; if so, fall back to "approve all" / "review individually" branching.
- **Render directive vocabulary lock-in.** Once an agent learns the `render: plan` directive, changing the schema is a coordination problem. v1.1 directives are versioned: `{"render": "plan", "version": 1, ...}`. Bridge accepts known versions, rejects unknowns. v2 can add `version: 2` schemas alongside.
- **How aggressively does the bridge intercept fenced blocks?** Strictly only ` ```switchboard` fenced blocks, never plain ` ``` ` code blocks (which the agent legitimately uses for code samples). Confirmed approach: the language tag is the discriminator.

**1c. Terse tool descriptions (no protocol dependency).**

When jcode emits a `ToolStart` event, v1 renders it as the bare tool name: "✓ Read", "⏳ Bash". This is uninformative when there are 5-10 tool calls in a turn. v1.1 surfaces a 4-8 word description per tool call so users can scan what the agent is actually doing.

Three sources for the description, in descending order of quality:

**Source 1: agent-supplied via directive.** The system-prompt addendum teaches the agent to populate a `description` field on each tool call. The bridge reads it from the agent's `ToolStart` payload (or, where the protocol doesn't carry it, from a fenced ` ```switchboard ` directive immediately preceding the tool call).

**Source 2: tool-name + first-arg heuristics.** Fallback for when the agent doesn't supply a description. A small lookup at `internal/render/toolfriendly.go` mapping ~15-20 known tools to a friendly verb + first interesting argument:

| Tool | Heuristic |
|------|-----------|
| `Read` + path | "Reading `auth.go`" |
| `Bash` + cmd | "Running `npm test`" (first 30 chars of cmd) |
| `Grep` + pattern | "Find `handleAuth` uses" |
| `Edit` + path | "Editing `bar.py`" |
| `Glob` + pattern | "Listing `*.tsx`" |
| `WebSearch` + query | "Search: <query>" |
| `WebFetch` + url | "Fetching <hostname>" |
| `Write` + path | "Writing `out.json`" |
| `MultiEdit` + path | "Editing `foo.go` (multi)" |
| `Task` + prompt | "Subagent: <first 4 words>" |
| (other) | "Running `<tool>`" |

The 15-20 mappings are tested against a corpus of real jcode tool calls — sample 50 from recent dogfood usage, write expected friendly descriptions for each, ship the function when it matches >90%. Corpus + expectations are committed as a regression test.

**Source 3: tool name only.** Last resort if neither directive nor heuristic apply: "Running BashOutput". Better than blank.

#### Style guide for descriptions

LLMs over-trained on prose will drift toward verbose descriptions ("I'll search for usages of authMiddleware in the auth/ subdirectory") that defeat the purpose. The system-prompt addendum addresses this with five concrete examples rather than abstract length guidance:

```
Tool descriptions: ≤8 words, telegraphic. No "I'll" / "Let me" / "Going to".
Use shorthand: paths instead of names, drop articles, prefer present-tense verbs.
Read aloud-friendly — avoid numerals-for-words ("4" for "for"); spelled
prepositions are short enough.

Verbose:                                     → Terse:
"Reading the configuration file"             → "Reading config.toml"
"Searching for authMiddleware uses"          → "Find authMiddleware uses"
"Running the test suite for auth module"     → "Running auth/ tests"
"Editing the settings file to add CORS"      → "Edit settings.py: add CORS"
"Fetching the GitHub PR description"         → "Fetching PR description"
```

**Soft cap, not hard cap.** The agent is asked for ≤8 words. The render layer truncates with ellipsis at 10 words, logging a warning. The truncation is cosmetic — better than rejecting the call. Frequent truncation is a signal that the prompt isn't holding and the examples need adjustment.

**Style applies to heuristics too.** The fallback table uses the same telegraphic style ("Reading X", "Find X uses", "Edit X", "Run X") so the visual texture stays consistent regardless of source.

#### Drift monitoring

LLM stylistic compliance drifts over conversations and across model versions. To catch this without manual eyeballing, the bridge tracks a quiet metric: `avg_words_per_tool_description` per session, written to the audit log on each `Done` event. If the rolling average across all sessions creeps above 7 words over a week, the system prompt examples need refreshing.

This is data-grounded rather than vibe-grounded — the kind of measurement nothing else (cc-connect included) does, and the only way to detect drift before users do.

### State and configuration additions

```toml
[render.descriptions]
target_words = 8                         # what the prompt asks for
hard_truncate_words = 10                 # cosmetic safety net
log_truncations = true                   # warn when agent exceeds target
log_drift_threshold = 7                  # avg above this triggers a warn
```

No schema changes. The metric is a column on `audit_log.payload_summary_json`'s tool entries (already JSON, so additive).

### Open questions for the design phase

- **What happens when a tool's first argument is itself long?** A 200-char shell command in `Bash` doesn't make a useful description. The heuristic truncates at 30 chars + ellipsis: "Running `npm test --watch --coverage --reporter=…`". Confirm threshold during dogfood.
- **Should descriptions be localised?** No, English-only for v1.1. Worth noting for v2 multi-locale support.
- **Where does the description live in the directive?** Inline on the `ToolStart` if jcode's protocol carries it; via a preceding ` ```switchboard {"render": "tool_describe", "description": "..."}` ``` block if not. The latter is uglier but works without a protocol change. v1.1 supports both; v1.2 may consolidate once the upstream protocol shape is settled.

### Estimated effort

1-2 days. System-prompt addendum + heuristic table + truncation + drift metric. Most of the work is curating the 50-call corpus and the 5 prompt examples; the code is straightforward.

## Feature 2: LLM-based notification router

### Mechanism

When a webhook event matches no `[[routes]]` entry in `[ingest]`, instead of going straight to the fallback channel, it goes through the LLM router:

1. Build a prompt containing:
   - The event's source, type, and a redacted summary of the payload (per the §13 audit allowlist — same fields that go into `audit_log.payload_summary_json`).
   - A list of currently active threads, last 24 hours, scoped to the v1.1 audit log: each entry has channel name, thread topic (first message excerpt), participants, last activity timestamp, and any registered correlation keys.
   - The active sessions' workdirs.
   - A clear instruction: "Which thread is this most likely associated with? Respond as JSON: `{thread_id: 'channel_id:thread_ts', confidence: 0-100, reasoning: 'brief'}`. If no thread is a good match, respond with `{thread_id: null, confidence: 0, reasoning: '...'}`."
2. Call Anthropic API with Haiku-class model (`claude-haiku-4-5` or whatever's smallest/cheapest at v1.1 time). Token budget: ~2-4k input, ~200 output. Cost: cents per day.
3. Parse response. Defensive — handle JSON parse failures, non-conforming responses, API timeouts. On any failure: route to fallback as if no rule matched.
4. If `confidence >= [routing.llm].confidence_threshold` (default 80): post to suggested thread with `_routed by AI (confidence X%, <reasoning>)_` italic footer.
5. If below threshold: post to fallback with the LLM's full reasoning attached as a Slack snippet for operator review.
6. On successful AI routing, automatically write a `thread_correlations` row so future events with the same correlation key route deterministically — i.e., the LLM teaches the rule engine over time.

### Schema additions

```sql
-- Track LLM routing decisions for audit and post-hoc accuracy review
CREATE TABLE llm_routing_decisions (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  webhook_inbox_id INTEGER NOT NULL,
  decided_at      INTEGER NOT NULL,
  model           TEXT NOT NULL,
  thread_id       TEXT,                  -- nullable; "no match" decisions
  confidence      INTEGER NOT NULL,
  reasoning       TEXT,
  posted_to       TEXT NOT NULL,         -- "suggested" or "fallback"
  user_feedback   TEXT,                  -- 'confirmed', 'rejected', null
  feedback_at     INTEGER,
  FOREIGN KEY (webhook_inbox_id) REFERENCES webhook_inbox(id)
);

CREATE INDEX idx_llm_routing_inbox ON llm_routing_decisions(webhook_inbox_id);
CREATE INDEX idx_llm_routing_feedback ON llm_routing_decisions(user_feedback, decided_at);
```

The `audit_log.routing_confidence` column added in v1 is now actively populated.

### User feedback loop

When an event is routed by the LLM and posted into a thread, the message includes a `context_actions` block at the bottom with native `feedback_buttons` (positive: "Right thread" / negative: "Wrong thread") plus an `icon_button` for "Move to fallback." Click handling is wired through the same interactivity URL as tool approvals (Feature 1b), routed by `action_id`:

- **Positive (👍 "Right thread"):** mark the routing as `confirmed` in `llm_routing_decisions`. Strengthens the case for auto-correlating going forward (in v1.2 or later).
- **Negative (👎 "Wrong thread"):** mark `rejected`, repost the message into the fallback channel with explanation, delete the original.
- **Move to fallback:** explicit "I see what you tried but this just belongs in #notifications" — same as negative but suppresses follow-up "would you like me to learn from this?" prompts.

Using native `feedback_buttons` rather than Slack reactions has two advantages: the action_id is structured (no parsing emoji intent), and the button labels are explicit ("Right thread" vs "👍" leaves users guessing what they're voting on). The `feedback_buttons` element is a first-class Slack block (`type: feedback_buttons` inside `context_actions`), supported in `slack-go/slack`.

This produces a clean audit trail of LLM accuracy without requiring human attention except where the decision was wrong.

### Configuration

```toml
[routing.llm]
enabled = true
model = "claude-haiku-4-5"
confidence_threshold = 80                # 0-100; below = fallback
max_input_tokens = 4000
include_thread_count = 30                # how many recent threads to show the LLM
api_key = "${ANTHROPIC_API_KEY_NOTIFICATION_ROUTING}"
# Cost guardrail: refuse to call LLM if past 30d cost > this (USD)
monthly_budget_usd = 5.0
```

The LLM router is *opt-in*. The default v1.1 deployment leaves it disabled until the operator wants to flip it on.

### Cost ceiling

Per-call cost on Haiku at expected token sizes is well under a cent. Even at 1000 webhooks/day all going through the LLM, monthly cost is small. The `monthly_budget_usd` guardrail exists so a misconfigured rule (or a Sentry-storm) can't run up unexpected costs — past the budget, the router silently degrades to fallback-channel-only and logs a warning.

### Open questions for the design phase

- **What threads count as "active"?** Last activity in 24h is a starting point; might need 7d for slow-paced workflows. Configurable per-source.
- **Do we include private/DM threads in the candidate list?** Probably no — LLM should not see content from threads it might route into accidentally. v1.1 limits LLM-router-eligible threads to channels with `[[channels]]` entries.
- **What if the LLM returns confidence 79 (just below)?** Hard threshold or ramp? Default to hard threshold; if it turns out to be useful, add a "post to suggested thread with strong caveat language" middle band in v1.2.
- **How do we evaluate accuracy over time?** The `llm_routing_decisions` table with feedback lets us run a query like "of the last 100 LLM decisions, how many got 'Wrong thread' clicked?". Worth building a `switchboard --routing-stats` command for operators.
- **Are confirms and rejects scored differently?** A confirm is signal but cheap — the message landed in the right place, you'd click "Right thread" even if the LLM was uncertain. A reject is rare and expensive. Weight rejects more heavily in any "how is the router doing" calculation.

### Estimated effort

5-8 days. Most of it is the prompt engineering and the feedback loop polish; the API call itself is trivial.

## Sequencing

Suggested order, regardless of v1.1 vs split releases:

1. **Block Kit rendering (1a)** first. Touches the coalescer and outbound layer, easier to test in isolation. No external blockers. Render-directive interceptor lands in the same milestone — same code path, same tests.
2. **Terse tool descriptions (1c)** alongside (1a). Reuses the directive interceptor; the heuristic table is independent code. Worth shipping with 1a so the rendering work has good descriptions from day one rather than retrofitting later.
3. **LLM router (2)** second. Independent of rendering work. Mostly new code paths, low risk to v1 behaviour. Uses the rendering layer's `feedback_buttons` support, which is implemented as part of (1).
4. **Block Kit interactive (1b)** last, *if* the jcode PR has landed. Otherwise defer to v1.2.

Total estimate: ~2-3 weeks for 1a + 1c + 2 if working in parallel; ~4 weeks if sequential. Add 1-2 weeks if 1b is included in v1.1. (Estimates dropped ~30% from rev 1 because the dedicated block types eliminate most custom layout work.)

## Test plan additions for v1.1

These get appended to the v1 test plan:

**§B1. Block Kit rendering correctness.** Visual regression-ish: post a known prompt with a known response, capture the rendered Block Kit JSON for each output type, diff against a golden file in `testdata/blockkit-golden/`. Goldens cover:
- `task_card` for live progress (with various tool counts)
- `plan` for multi-step dispatches (rendered from `render: plan` directive)
- `markdown` for completion reports (with task-list checkboxes)
- `card` for webhook notifications
- `card` with `hero_image` for image-bearing notifications
- `alert` at all three levels (success/warning/error)
- `deep_research_result` shape (section + markdown + carousel) for citation-bearing briefs

**§B1a. Render-directive interceptor.** Send the agent stream a `TextDelta` containing a fenced ` ```switchboard ` JSON block. Confirm:
- Recognised directive (`plan`, `brief`, `poll`, `tickets`, `todos`) is intercepted, validated, removed from visible text, and replaced with rendered blocks.
- Unknown directive name (e.g., `render: invented`) falls through as visible code block (with `strict_directive_validation = false`).
- Invalid schema for known directive (e.g., `render: plan` missing `tasks`) falls through as visible code block.
- Plain ` ``` ` code blocks (no language tag, or `python`/`go`/etc.) are NOT intercepted; agent's normal code samples render as code as expected.
- Versioned directive (`{"render": "plan", "version": 1, ...}`) parses; unknown version produces an error block.

**§B1b. Block-type fallback strategy.** For each rendered block, confirm `text` parameter is set in the `chat.postMessage` call with a one-line plain summary. Test by manually clearing the bot's render in a test workspace (e.g., using a Slack client without the latest block support — older mobile, accessibility tools): the fallback string should be informative and the conversation should still be followable.

**§B1c. Tool-friendly descriptions — agent-supplied path.** With the system-prompt addendum applied, run a session with 20 tool calls. Sample 10 of them and verify:
- Description is present.
- Description is ≤10 words (truncation should not have fired).
- Style matches the addendum: telegraphic, no "I'll"/"Let me", paths-not-names, present-tense.
- Description matches the agent's actual intent (eyeball judgement).
Acceptance: ≥80% match expectations on first pass. Below that, refine prompt examples and re-run.

**§B1d. Tool-friendly descriptions — heuristic fallback.** Force the agent to omit descriptions (use a model without the addendum, or strip descriptions in the bridge for testing). Verify each tool call gets a heuristic description from the lookup table. Sample 20 calls covering Read, Bash, Grep, Edit, Glob, WebSearch, Write, MultiEdit, Task. Acceptance: ≥90% match expected friendly form. Bash commands >30 chars truncate with ellipsis.

**§B1e. Description drift monitoring.** Run 50 tool calls. Inspect `audit_log` for the per-session `avg_words_per_tool_description`. Verify the metric is populated. Manually inflate descriptions to >7 average words for a session; verify the warn-level log line fires.

**§B2. Tool approval flow** (gated on 1b shipping).
- Single-tool approval: post the `approval` shape (card with Allow/Deny buttons). Approval prompt appears within 1s of jcode emitting the request.
- Click Allow → `ToolApprovalResponse` sent to jcode within 1s; agent proceeds; message edits to "✓ Allowed by @leigh at HH:MM" via `chat.update`.
- Click Deny → tool execution skipped; agent receives denial; message edits accordingly.
- Click Allow all → grant persisted in `session_tool_grants`; same tool in same session skips prompt thereafter.
- Multi-tool approval (carousel): jcode emits ≥ `allow_carousel_threshold` pending approvals; bridge posts an `approval_with_carousel` shape with one card per tool call. User swipes/scrolls through and approves each independently.
- Carousel completion: when all carousel cards are resolved, the message edits to a summary "✓ 3 of 4 approved, 1 denied — agent proceeding."
- Bridge restart mid-approval: kill the bridge between the approval message being posted and the user clicking; restart; click. Confirm `pending_approvals` row found, response routed to jcode correctly.
- Multi-user race: two users click different buttons within milliseconds; first wins (verify in `pending_approvals.resolved_by`); second gets ephemeral "this was already resolved by @other-user" message.
- Timeout: no click for 10 min → status updates to `expired`, message edits to "⏰ Auto-denied after 10 min", agent receives denial.

**§B3. LLM router accuracy.** Manually craft 20 webhook events: 10 with a clear "right thread" and 10 with no good match. Run through the router. Tally accuracy. Confirm:
- Clear matches: confidence ≥ 80, routed to the expected thread.
- No-match cases: confidence < 80, routed to fallback.
- Bad JSON / API timeout / rate limit: degrades gracefully to fallback.

**§B4. LLM cost guardrail.** Set `monthly_budget_usd` artificially low. Trigger 50 webhooks. Confirm router stops calling Anthropic mid-run, logs a warning, and degrades to fallback.

**§B5. Feedback loop with native buttons.** Send a webhook through LLM router; the message should include a `context_actions` block with feedback buttons. Click "Right thread" → `llm_routing_decisions.user_feedback = 'confirmed'`. Click "Wrong thread" on another routed message → message reposted to fallback channel, original deleted, decision marked `rejected`. Click "Move to fallback" → same as wrong-thread but no follow-up prompt.

**§B6. `markdown` block long-content handling.** Generate an agent response > `markdown_split_chars` (default 8000). Confirm output splits into multiple threaded messages cleanly at logical boundaries (paragraph breaks, never mid-sentence). No truncation.
